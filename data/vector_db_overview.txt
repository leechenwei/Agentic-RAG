Vector Databases

A vector database is a specialized data store designed to index and search
high-dimensional vector embeddings. Each record typically consists of a vector
(an array of floating-point numbers produced by an embedding model) plus
associated metadata and optionally the original text or file reference.

The core operation is approximate nearest neighbor (ANN) search: given a query
vector, find the k records whose vectors are closest under a chosen distance
metric. Common metrics are cosine similarity, Euclidean (L2) distance, and dot
product. Exact nearest neighbor search scales poorly, so production systems use
ANN indexes such as HNSW (Hierarchical Navigable Small World graphs), IVF
(Inverted File index), or product quantization. HNSW offers excellent recall
and latency trade-offs and is the default in most modern vector databases.

Popular vector databases include ChromaDB, Pinecone, Qdrant, Weaviate, Milvus,
and pgvector (a PostgreSQL extension). ChromaDB is a lightweight embedded
database written in Python that persists to local disk via SQLite and Parquet,
making it ideal for prototypes and small-to-medium workloads. Pinecone is a
fully managed cloud service with strong horizontal scaling. Qdrant is open
source with a Rust core, supports filtering on metadata, and can be self-hosted
or used via Qdrant Cloud. pgvector lets teams add vector search to an existing
Postgres deployment without introducing a new system.

When selecting a vector database, consider operational model (embedded vs
client-server vs managed), maximum corpus size, query latency requirements,
metadata filtering capabilities, support for hybrid search, and cost. For
prototypes ChromaDB is usually the fastest path; for production with millions
of vectors and multi-tenant access, Qdrant, Weaviate, or Pinecone are the
common choices.
